# AURA Sprint 278 — Bounded ORION Capture, Application, File, and OBS Actions

## Status

Sprint 278 activates a bounded-action **runtime contract** for the eight
action types approved by Sprint 277. It does not make the ATLAS host execute
Windows actions and it does not expose a general-purpose execution CLI.

The component is:

- package: `aura/orion_bounded_action_runtime`
- manager: `AuraOrionBoundedActionRuntimeManager`
- component version: `0.1.0`
- product version: `v1.3.1`
- sprint: `278`

## Public action catalog

The exact action catalog remains:

1. `capture_single_screenshot`
2. `capture_selected_window`
3. `open_allowlisted_application`
4. `create_controlled_file`
5. `create_controlled_folder`
6. `obs_start_recording`
7. `obs_stop_recording`
8. `obs_switch_scene`

No generic shell, arbitrary command, raw executable, unrestricted path, or
raw OBS endpoint action is introduced.

## Authorization composition

The runtime composes only through public Sprint 277 APIs:

- `verify_audit_chain`
- `validate_permission`
- `consume_permission`
- `record_execution_outcome`
- `audit_events`

A request is built from an in-process Sprint 277 permission. Its target and
parameter digests must exactly match the approved preview and permission.
Execution cannot begin until the signed, single-use permission-consumption
envelope has produced an authorization receipt with the same pairing,
device, live-link session, capability, preview, target, parameter, and
permission bindings.

The runtime signs its post-adapter execution outcome through the public
pairing API and records that outcome through Sprint 277. Private runtime APIs
are not used.

## Adapter boundary

The core manager is transport-agnostic and platform-agnostic. It imports no
Windows API, screen-capture library, OBS client, or process launcher.

Three adapter forms are defined:

- `AuraNonExecutingOrionBoundedActionAdapter`: default, fail-closed, and
  incapable of real execution.
- `AuraFakeOrionBoundedActionAdapter`: isolated test adapter that may write
  only inside temporary acceptance roots.
- `AuraWindowsOrionBoundedActionAdapter`: guarded adapter for explicit ORION
  injection.

The Windows adapter owns optional platform dependencies and process calls.
It requires Windows, an explicit enabled flag, and injected capture callbacks
or an injected OBS controller where applicable. Application launch uses an
absolute path resolved from a logical allowlist ID and calls
`subprocess.Popen` with `shell=False`.

## Capture constraints

Capture output is generated by the manager inside one configured absolute
capture root. Callers cannot provide an output path.

- output format is PNG
- overwrite is disabled
- symlink traversal is rejected
- path escape is rejected
- byte size is bounded
- artifact path, byte count, and SHA-256 digest are verified
- selected-window capture requires exact `window_id` and
  `window_title_digest` binding
- selected-window capture never falls back silently to the desktop

Raw capture bytes are never stored in action history.

## Application constraints

Application requests use a logical application ID. The ID must map to an
absolute, non-symlink executable path in local configuration.

The following are forbidden:

- raw executable supplied by the request
- raw command supplied by the request
- command-line arguments supplied by the request
- `shell=True`
- string shell invocation

Action history stores only a redacted result and digests.

## Controlled file and folder constraints

File and folder actions are confined to one configured absolute root.

- request paths must be relative
- `..` traversal is rejected
- absolute paths are rejected
- symlink escape is rejected
- overwrite is disabled
- file content is size-bounded
- UTF-8 is the only initial controlled text encoding
- artifact size and SHA-256 digest are verified
- partial adapter failures are recorded as bounded failures

Raw file content is not written to action history.

## OBS constraints

OBS operations require a dedicated injected adapter/controller. The core
manager creates no OBS client and opens no network connection.

- recording start and stop use fixed logical actions
- scene switching uses a logical allowlisted scene ID
- raw endpoint, password, and scene-name input are forbidden
- OBS connection profile selection comes from local configuration
- OBS failures are redacted before they enter history or audit outcome notes

## Lifecycle and failure behavior

The runtime states are:

- `idle`
- `validating`
- `authorized`
- `executing`
- `succeeded`
- `failed`
- `outcome_unconfirmed`

The runtime is single-flight.

A successful or bounded failed action records the Sprint 277 outcome and
returns to safe-idle. If the adapter may have acted but the Sprint 277 outcome
cannot be recorded, the runtime enters `outcome_unconfirmed`. New actions and
ephemeral reset remain blocked there. Sprint 279 owns watchdog, emergency
stop, and recovery/reconciliation for that state.

## Persistence and history

Sprint 278 action requests, raw parameters, and execution history are
in-memory only. Sprint 277 remains the durable append-only audit source.

History is bounded and redacted. It contains no:

- raw capture bytes
- raw file content
- raw command or executable input
- OBS credential
- pairing secret
- general-memory handoff

## CLI boundary

Only read-only commands are exposed:

- `orion-bounded-action-status`
- `orion-bounded-action-inspect`
- `orion-bounded-action-self-test`

There is intentionally no execution CLI in Sprint 278.
